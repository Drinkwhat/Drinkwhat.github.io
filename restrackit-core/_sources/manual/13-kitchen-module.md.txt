# 🍳 Kitchen Module Documentation

## 1. Overview

The **Kitchen Module** (`app/modules/kitchen`) is responsible for managing culinary operations within the restaurant. It allows chefs and kitchen staff to:
- Define standard **Recipes** (Ricette) with specific ingredients.
- Check **Ingredient Availability** in real-time based on active inventory batches.
- Record **Preparations** (Preparazioni), linking them to the specific batches used for traceability.

This module ensures that every dish prepared can be traced back to the specific lots of raw materials used, maintaining food safety standards and inventory accuracy.

---

## 2. Data Models

### 📖 Recipe (`recipes`)
Defines a standard culinary preparation.
- `id`: Unique identifier.
- `store_id`: Links the recipe to a specific store/restaurant.
- `name`: Name of the dish (e.g., "Fajitas").
- `description`: Optional instructions or notes.
- **Relationships**:
  - Has many `RecipeIngredient`.
  - Has many `Preparation`.

### 🥕 RecipeIngredient (`recipe_ingredients`)
Links a recipe to a product from the master data.
- `recipe_id`: Parent recipe.
- `product_master_id`: The ingredient product (e.g., "Chicken Breast").
- `standard_quantity`: Default amount used.
- `unit`: Unit of measure (e.g., "kg", "l", "pcs").

### 👨‍🍳 Preparation (`preparations`)
Represents the execution of a recipe at a specific time.
- `id`: Unique identifier.
- `recipe_id`: The recipe being prepared.
- `operator_id`: The chef/staff member who performed the preparation.
- `created_at`: Timestamp of the preparation.
- `notes`: Specific notes for this batch.
- **HACCP Integration**: Every preparation automatically creates a new **Batch** in the inventory for the resulting product.
- **Relationships**:
  - Has many `PreparationItem`.
  - Links to a `Batch` record in the inventory.

### 📦 PreparationItem (`preparation_items`)
The specific link between a preparation and the inventory batches used as ingredients.
- `preparation_id`: Parent preparation.
- `batch_id`: The specific inventory lot used (e.g., Lot "L123" of Chicken).
- `quantity_used`: Amount actually used from this batch.
- `unit`: Unit of measure.

## 3. HACCP Compliance & TMC Calculation

The module enforces strict food safety tracking through the **Temperature Monitoring Certificate (TMC)** / Expiry logic.

### 📋 Mandatory Conservation Method
Every **Recipe** must be linked to a `ConservationMethod`. This method defines:
- `trigger_event`: Usually `PRODUCTION_END` for preparations.
- `duration_days` and `duration_hours`: The shelf life of the preparation.

### 🧮 Expiry Date Calculation (TMC)
When a preparation is recorded, the system automatically calculates the expiry dates for the new lot:

1.  **Primary Expiry (`expiry_date`)**: 
    - Calculated as the **minimum** of the expiry dates of all ingredients used.
    - If no ingredients have an expiry date, it defaults to tomorrow.
2.  **Secondary Expiry (`secondary_expiry_at`)**:
    - Calculated by adding the recipe's `ConservationMethod` duration to the current time.
    - **Capping**: The secondary expiry is automatically capped at the end of the primary expiry date to ensure safety.

## 4. Semilavorati (Pre-preparations)

Recipes in Restrackit are intrinsically linked to the inventory via the **Semilavorato** concept:
- When a recipe is created, a corresponding `ProductMaster` is automatically generated with the category **"Semilavorato"**.
- This product represents the "finished dish" or "pre-preparation" that can be stored in the fridge.
- Inventory searches (via AI or API) can filter out these items to avoid cluttering raw material searches.

## 5. API Endpoints

All endpoints are prefixed with `/kitchen` and require authentication.

### Recipes Management
- **POST** `/kitchen/recipes`: Create a new recipe (requires `conservation_method_id`).
- **GET** `/kitchen/recipes`: List all recipes for the current store.
- **GET** `/kitchen/recipes/{id}`: Get details of a specific recipe.
- **PATCH** `/kitchen/recipes/{id}`: Update a recipe (supports optimistic locking via `version`).

### Availability Check
- **GET** `/kitchen/recipes/{id}/availability`:
  - Returns the list of ingredients for a recipe.
  - For each ingredient, lists the **Active Batches** currently in stock.

### Preparation Recording
- **POST** `/kitchen/preparations`: Record a new preparation.
  - **Auto-Inventory**: Creates a new `Batch` for the recipe's product.
  - **TMC Logic**: Calculates primary and secondary expiry dates.
  - **Validation**: Verifies that ingredients batches are active.
- **GET** `/kitchen/preparations`: List preparation history with filters.

## 6. Usage Flow

### Step 1: Define a Recipe
```json
POST /kitchen/recipes
{
  "name": "Tomato Sauce (Base)",
  "conservation_method_id": 5,
  "ingredients": [
    {"product_master_id": 101, "standard_quantity": 5.0, "unit": "kg"}
  ]
}
```

### Step 2: Check Availability
Before preparing, the chef checks which ingredient lots are in stock.
```http
GET /kitchen/recipes/1/availability
```
**Response:**
```json
{
  "recipe_id": 1,
  "ingredients": [
    {
      "product_name": "Tomatoes",
      "active_batches": [
        {"id": 501, "lot_code": "LOT-A-2023", "expiry_date": "2023-12-31"},
        {"id": 502, "lot_code": "LOT-B-2024", "expiry_date": "2024-01-15"}
      ]
    }
  ]
}
```

### Step 3: Record Preparation
```json
POST /kitchen/preparations
{
  "recipe_id": 1,
  "operator_id": "chef_mario",
  "items": [
    {"batch_id": 501, "quantity_used": 5.0, "unit": "kg"}
  ]
}
```
**Result**: A new lot `PREP-YYYYMMDD-HHMMSS-ID` is created in the inventory, with scrupulously calculated expiry dates.

---

## 7. Permissions

- `kitchen:read`: View recipes and preparations.
- `kitchen:manage_recipes`: Create/Update recipes.
- `kitchen:write_preparation`: Record new preparations.
