# Architecture Overview

## System Design

Restrackit is a FastAPI-based REST API for restaurant inventory management with AI-powered image analysis, designed using a **Modular Monolith** architecture.

```text
┌─────────────────────────────────────────────────────────────┐
│                      Client (App/Web)                       │
└────────────────────────┬────────────────────────────────────┘
                         │ HTTP/REST (JSON)
                         ▼
        ┌────────────────────────────────────┐
        │      FastAPI Application           │      ┌──────────────┐
        │  (app/main.py)                     │◄────►│    Redis     │
        └────────────┬───────────┬───────────┘      │(Cache/Queue) │
                     │           │                  └──────────────┘
    ┌────────────────┼───────────┼────────────────────────┐
    │                │           │                        │
    ▼                ▼           ▼                        ▼
┌──────────┐   ┌──────────┐ ┌─────────────┐        ┌─────────────┐
│ Vision   │   │ Gemini   │ │ PostgreSQL  │        │ arq Worker  │
│ API      │   │ Agent    │ │ Database    │        │ (Async Task)│
│(OCR)     │   │ (AI)     │ │ (Data)      │        │             │
└──────────┘   └──────────┘ └─────────────┘        └─────────────┘
```

## Technology Stack

### Backend

- **Framework**: FastAPI (Python >= 3.14)
- **ORM**: SQLAlchemy 2.0 (Sync/Async ready)
- **Migrations**: Alembic
- **Linter**: Ruff (replaces Flake8/Black/Isort)

### AI & Data

- **OCR**: Google Cloud Vision API
- **LLM**: Google Gemini 2.0 Flash (via `google-genai` SDK)
- **Database**: PostgreSQL 16 (Alpine)
- **Cache & Queue**: Redis 7.2 (with msgpack serialization)
- **Task Queue**: arq (Async Redis Queue)

### DevOps

- **Container**: Docker (Multi-stage: Builder -> Runtime)
- **CI/CD**: GitHub Actions (Lint, Test, Build, Docs)
- **Testing**: Pytest + Coverage (100% target)
- **Monitoring**: Sentry (Error Tracking) + Apitally (API Analytics)

## Key Components

### 1. Modular Architecture (`app/modules/`)

Each module (`inventory`, `products`) is self-contained with its own:

- **Router**: Defines API endpoints
- **Service**: Implements business logic
- **Models**: Defines DB tables
- **Schemas**: Defines DTOs (Data Transfer Objects)

### 2. Error Handling (`app/core/errors.py`)

Standardized error handling using `AppError` and specific codes (`ERR_...`):

- All errors return a uniform JSON structure.
- Unhandled exceptions are caught by global middleware and returned as 500 System Errors.

### 3. AI Agent (`app/services/ai_agent.py`)

A specialized service that:

1. Receives raw OCR text.
2. Uses a custom tool (`search_product_tool`) to query the DB for product matches.
3. Returns structured data (Product, Lot, Expiry, Confidence).

### 4. Caching System (`app/shared/cache.py`)

Implementation of an asynchronous caching layer:
- **Decorator `@cached`**: Applied to service methods (including `products` and `kitchen`) to cache results in Redis.
- **Serialization**: Efficient binary serialization using `msgpack` with custom encoders for `date`/`datetime`.
- **Model Hydration**: Automatically reconstructs Pydantic models from cache hits.
- **Fail-Open**: If Redis is unavailable, the system transparently falls back to database queries, ensuring high availability.
- **Selective Invalidation**: Cache entries are automatically invalidated on resource updates or deletions using key pattern matching.

### 5. Background Tasks (`app/worker.py`)

Asynchronous task processing using `arq`:
- **Singleton Pool**: Uses a persistent Redis connection pool to minimize overhead.
- **Msgpack Serialization**: Custom serializers for `arq` that handle `date`/`datetime` objects via ISO conversion.
- **Status Mapping**: Detailed job tracking that distinguishes between `arq` execution state and internal task result status (e.g., catching AI failures).
- **Offloading**: Moves heavy OCR and AI analysis to a separate worker process.
- **Shared Storage**: Uses Docker volumes (`uploads_data`) for temporary file processing between the API and workers.
- **Monitoring**: Real-time status retrieval via `/inventory/analyze/{job_id}`.

### 6. Allergen Management (`app/shared/allergens.py`)

A specialized bitmask-based system for EU-compliant allergen tracking:
- **Bitmask Storage**: Allergens are stored as a single integer (bitmask) in the `ProductMaster.allergen_mask` column, ensuring extreme performance for filtering.
- **I18n Support**: Dedicated mapping endpoints (`/products/allergens`) provide localized labels (IT/EN) for frontend display.
- **Strict Typing**: Uses `AppAllergen` Enum to prevent invalid data entry and ensure consistency across the API.
- **Filtering**: Efficiently supports queries like "gluten-free" or "free from X, Y, Z" using bitwise operators in the database.

## Data Flow: Image Analysis

1. **Upload**: Client sends image to `POST /inventory/analyze`.
2. **OCR**: `OCRService` calls Google Vision to get text.
3. **Agent**: `InventoryAgentService` sends text + store context to Gemini.
4. **Prompt Injection Protection**: Untrusted OCR text is wrapped in delimiters (`[START_OCR]...[END_OCR]`) and isolated using **System Instructions**.
5. **Structured Output**: Enforces valid JSON results via **JSON Schema Mode** and Pydantic models.
6. **Tool Use**: Gemini calls `search_product_tool` to validate product name against `products_master` table.
7. **Response**: Structured JSON is returned to the client for confirmation.
8. **Confirm**: Client sends validated data to `POST /inventory/confirm` to save in `batches` table.

## Multi-Tenancy & Security

### 1. Row Level Security (RLS)
The system implements a dual-layer isolation strategy:
- **Application Layer**: All service queries are scoped by `store_id` (e.g., `filter(ProductMaster.store_id == store_id)`).
- **Database Layer**: PostgreSQL Row Level Security is enabled on core tables. The application uses a `SET LOCAL app.current_store_id` command within every transaction to enforce isolation at the database level, preventing data leaks even in case of application bugs.

### 2. Authentication (Keycloak OIDC)
- **JWT Validation**: Real-time token validation against Keycloak JWKS with localized caching in Redis for performance.
- **Granular Permissions**: A custom `RequirePermission` dependency maps Keycloak roles to fine-grained action permissions (e.g., `INVENTORY_WRITE`).

### 3. Monitoring & Observability
- **Request Tracing**: Every request is assigned a unique `X-Request-ID`, propagated through logs, database contexts, and external service calls for end-to-end tracing.
- **Sentry**: Captures unhandled exceptions and performance bottlenecks with release tracking tied to Git commit hashes.
- **Apitally**: Provides real-time insights into API usage, response times, and error rates without impacting performance.
