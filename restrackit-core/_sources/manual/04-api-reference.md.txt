# API Reference

Questa pagina riassume gli endpoint principali. Per la lista completa e sempre aggiornata, usa:

- Swagger UI: `GET /docs`
- OpenAPI JSON: `GET /openapi.json`

> **Ambienti**: `/docs`, `/redoc` e `/openapi.json` sono esposti solo in ambienti non-production (`NON_PRODUCTION_ENVIRONMENTS`, default-deny per OWASP API9, vedi `app/main.py::resolve_docs_urls`). In produzione questi path restituiscono 404.

> **Schema versionato**: ogni [GitHub Release](../.github/workflows/release.yml) allega uno snapshot di `openapi.json` generato da `scripts/export_openapi.py` (`make export-openapi`). Usalo per generare client SDK (es. Dart/Flutter) puntati a una versione stabile del backend, invece di dipendere dal live endpoint di un ambiente dev/staging in continuo movimento. Dettagli: [07-cicd-pipeline.md](07-cicd-pipeline.md).

> **API IDs**: Tutti i path parameter usano UUID opachi (`public_id`) invece di interi sequenziali. Questo previene attacchi IDOR per enumerazione. Il campo `id: int` rimane nelle response per compatibilità e sarà rimosso in una versione futura.

> **Prefisso di versione**: tutti gli endpoint applicativi documentati di seguito sono
> esposti sotto il prefisso `/v1` (es. `POST /v1/inventory/analyze`). Gli endpoint
> infrastrutturali (`/health`, `/health/ready`, `/docs`, `/openapi.json`, `/ui/*`)
> restano invariati e non sono versionati.

## Authentication

La sicurezza è basata su Keycloak (OIDC) con token JWT:

```bash
TOKEN="<bearer-token>"
curl -H "Authorization: Bearer ${TOKEN}" http://localhost:8000/health
```

Se un utente ha accesso a più store, può selezionare lo store target tramite:

```bash
curl -H "Authorization: Bearer ${TOKEN}" -H "x-target-store: 1" http://localhost:8000/v1/inventory/list
```

Dettagli: [09-auth.md](09-auth.md)

## Core Headers (Tracing & Multi-tenancy)

Per garantire tracciabilità e isolamento dei dati, il sistema utilizza i seguenti header:

| Header | Descrizione | Obbligatorio |
|--------|-------------|--------------|
| `Authorization` | Bearer token JWT da Keycloak. | Sì |
| `X-Request-ID` | ID univoco della richiesta per il tracing. Se non fornito, viene generato dal server. | No |
| `X-Target-Store` | ID dello store su cui operare (necessario se l'utente ha accesso a più store). | Condizionale |

## Inventory Module (`/v1/inventory`)

### `POST /v1/inventory/analyze`

Analizza una foto di etichetta con OCR + Agent.

**Request** (multipart/form-data):

```bash
curl -H "Authorization: Bearer ${TOKEN}" \
  -F "file=@label.jpg" \
  "http://localhost:8000/v1/inventory/analyze?background=false"
```

**Response** (200 OK - Synchronous):

```json
{
  "status": "revision",
  "data": {
    "prodotto": "latte intero",
    "lotto": "L12345",
    "scadenza": "2024-12-31",
    "confidence": 0.95,
    "is_corrected": false
  },
  "warning": null,
  "raw_ocr_debug": "LATTE INTERO L12345 SCAD 31/12/24"
}
```

**Response** (200 OK - Asynchronous):

```json
{
  "status": "queued",
  "job_id": "84d72023-..."
}
```

**Errors**:

- `400 Bad Request`: File vuoto o dati non validi.
- `422 Unprocessable Entity`: Parsing AI fallito.
- `502 Bad Gateway`: Errore OCR/servizio esterno.

---

### `POST /v1/inventory/parse-text`

Analizza testo OCR già estratto lato client (es. ML Kit on-device), senza
inviare l'immagine. Restituisce lo stesso formato di `/v1/inventory/analyze`
in modalità sincrona.

**Request**:

```bash
curl -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"raw_text": "LATTE INTERO L12345 SCAD 31/12/24"}' \
  "http://localhost:8000/v1/inventory/parse-text"
```

**Response** (200 OK):

```json
{
  "status": "revision",
  "data": {
    "prodotto": "latte intero",
    "lotto": "L12345",
    "scadenza": "2024-12-31",
    "confidence": 0.95,
    "is_corrected": false
  },
  "warning": null,
  "raw_ocr_debug": null
}
```

**Errors**:

- `400 Bad Request`: `raw_text` mancante, vuoto, o oltre 5000 caratteri.
- `422 Unprocessable Entity`: Parsing AI fallito.
- `429 Too Many Requests`: Rate limit superato (10 richieste/60s).
- `502 Bad Gateway`: Errore del servizio Gemini.

### `POST /v1/inventory/confirm`

Confirm and save a validated batch into the database.

**Idempotency**: accepts an optional `Idempotency-Key` header. If a request
with the same key was already completed for the store within the last
`idempotency_key_ttl_hours` (default 24h), the endpoint returns the original
`201` response instead of creating a duplicate batch. If a request with the
same key is still being processed (a genuine concurrent duplicate), the
endpoint returns `409` with error code `ERR_BUS_009`. If the same key is
reused with a request body that differs from the one it was first associated
with, the endpoint returns `409` with error code `ERR_BUS_010` instead of
silently replaying the original response. Intended for clients on unreliable
networks (e.g. the mobile app) that may retry a request whose response was
lost in transit. The idempotency reservation and the batch write are
committed atomically, so a request either fully succeeds (batch created and
reservation completed) or fully fails (no orphaned "in progress" reservation
blocking a later retry).

**Request**:

```json
POST /v1/inventory/confirm
Content-Type: application/json
Authorization: Bearer <token>
Idempotency-Key: <client-generated-uuid>

{
  "product_name": "latte intero",
  "lot_code": "L12345",
  "expiry_date": "2024-12-31",
  "confidence": 0.95
}
```

**Response** (201 Created):

```json
{
  "status": "success",
  "message": "Lotto registrato con successo",
  "batch_id": 42,
  "product_name": "latte intero",
  "lot_code": "L12345",
  "expiry_date": "2024-12-31",
  "current_storage_method_id": null,
  "storage_expiry_at": null,
  "store_id": 1,
  "operator_id": "chef_marco",
  "confidence": 0.95,
  "created_at": "2024-02-05T10:00:00Z",
  "updated_at": "2024-02-05T10:00:00Z",
  "deleted_at": null,
  "version": 1
}
```

### `GET /v1/inventory/list`

List batches for the current store with optional filters.

**Request**:

```http
GET /v1/inventory/list?page=1&page_size=20&expired=false
Authorization: Bearer <token>
```

**Query Parameters** (principali):

- `page`: Numero pagina (default: 1)
- `page_size`: Elementi per pagina (default: 20)
- `product_name`: Filter by product name (partial match)
- `is_empty`: Filter by empty status (true/false)
- `expired`: Filter by expiration status (true=expired)

**Response** (200 OK):

```json
{
  "status": "success",
  "count": 1,
  "items": [
    {
      "batch_id": 42,
      "product_name": "latte intero",
      "lot_code": "L12345",
      "expiry_date": "2024-12-31",
      "store_id": 1,
      "operator_id": "chef_marco",
      "confidence": 0.95,
      "created_at": "2024-02-05T10:00:00Z",
      "is_empty": false
    }
  ]
}
```

### `POST /v1/inventory/batch/{batch_id}/move-storage`

```json
}
```

---

### `POST /v1/inventory/batch/{batch_id}/complete`

Segna un lotto come **concluso/esaurito**. Valorizza `batches.finished_at` e crea un record append-only in `batch_events`.

Documentazione completa del flusso (preparazione + lotto): [18-production-completion.md](18-production-completion.md)

Nota: richiede un body con `reason` (motivo standardizzato) e `notes` opzionali.

---

### `GET /v1/inventory/batch/{batch_id}/events`

Elenca lo storico eventi append-only del lotto (completamento preparazione, completamento lotto, ecc.).

### `GET /v1/products/list`

List all products for the store.

**Request**:

```http
GET /v1/products/list?gluten_free=true&allergen_free=nuts,peanuts
Authorization: Bearer <token>
```

**Query Parameters**:

- `category`: Filtra per categoria (es. "Latticini").
- `include_preparations`: Se `true`, include anche i semilavorati (default `false`).
- `gluten_free`: Se `true`, restituisce solo prodotti senza glutine.
- `allergen_free`: Lista di slug allergeni (separati da virgola) che il prodotto **NON** deve contenere.

**Response** (200 OK):

```json
{
  "status": "success",
  "count": 2,
  "items": [
    {
      "id": 101,
      "name": "Latte Intero",
      "category": "Latticini",
      "store_id": 1
    }
  ]
}
```

## Setup Module (`/v1/setup`)

> **Solo admin.** Tutti gli endpoint `/v1/setup` richiedono il permesso `ADMIN_ALL`
> (ruolo `admin`) e sono disabilitati in ambiente `production` (rispondono `404`).

### `POST /v1/setup/seed-demo`

Popola lo store demo (`id=1`) con un dataset completo e testabile in un'unica
chiamata: cataloghi (categorie, unità di misura, metodi di conservazione),
prodotti con regole di conservazione, lotti a scadenza futura/imminente/passata
e vuoti, ricette con ingredienti, preparazioni eseguibili e notifiche.

**Query params**:

- `reset` (bool, default `false`) — se `true` cancella lo store demo e lo
  rigenera da zero (stato pulito ad ogni esecuzione). Senza `reset` l'operazione
  è idempotente e non reinserisce dati se lo store esiste già.

**Request**:

```http
POST /v1/setup/seed-demo?reset=true
Authorization: Bearer <admin_token>
```

**Response** (200 OK):

```json
{
  "message": "Store demo popolato con dataset completo.",
  "store_id": 1,
  "reseeded": true,
  "counts": {
    "categories": 8,
    "units_of_measure": 2,
    "storage_methods": 3,
    "products": 15,
    "batches": 30,
    "recipes": 3,
    "recipe_ingredients": 11,
    "preparations": 4,
    "preparation_items": 12,
    "notifications": 5
  }
}
```

Se chiamato senza `reset` su store già popolato risponde `200` con
`"reseeded": false`. Per chef/manager risponde `403`; in `production` risponde `404`.

## Common Error Codes

Errors follow the standard schema:

```json
{
  "error": {
    "code": "ERR_VAL_001",
    "message": "Dati di input non validi: ...",
    "severity": "warning",
    "action": "Correggi i dati inviati secondo la documentazione API.",
    "details": null
  },
  "request_id": "..."
}
```

La mappatura HTTP status è definita dal catalogo errori applicativo: {mod}`app.core.error_catalog`.

Per l’elenco completo dei codici: [ERROR_CODES.md](ERROR_CODES.md)
