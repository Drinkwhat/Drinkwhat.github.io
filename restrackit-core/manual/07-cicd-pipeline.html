CI/CD Pipeline

Automated testing, building, and deployment with GitHub Actions.

Overview

Workflows in this repository handle testing, building, docs publishing, and releases:

Code Push/PR
    ↓
┌───────────────────────────────────────┐
│   Workflow: CI (ci.yml)               │
│   ├─ Quality (ruff + bandit)          │
│   ├─ SAST (semgrep)                   │
│   ├─ Unit Tests (pytest)              │
│   ├─ Coverage Reports (codecov)       │
│   └─ Quality Scan (sonarqube)         │
└───────────────────────────────────────┘
    ↓ (push main/dev + tags + PR)
┌───────────────────────────────────────┐
│   Workflow: Docker Build (docker-build.yml) │
│   ├─ Build Image                      │
│   ├─ Tag Version                      │
│   └─ Push to ghcr.io                  │
└───────────────────────────────────────┘
┌───────────────────────────────────────┐
│   Workflow: Docs (docs.yml)           │
│   ├─ Build Sphinx docs                │
│   └─ Publish to personal Pages repo   │
└───────────────────────────────────────┘
    ↓ (on CI success)
┌───────────────────────────────────────┐
│   Workflow: Release (release.yml)     │
│   ├─ Bump version (dev)               │
│   └─ Tag + Release (main)             │
└───────────────────────────────────────┘

CI Workflow (ci.yml)

Runs on every push and pull request to main and dev (with paths filters).

CI Jobs (current)

  • quality: fast gate (ruff check, ruff format --check, bandit)

  • semgrep: SAST scan (runs in parallel to quality/test)

  • secrets: secrets scan in git history (Gitleaks) to prevent leaking API keys/tokens/certs

  • test: unit tests + coverage + Codecov upload (needs quality)

  • sonarqube: static analysis and quality gate scan (needs test)

  • autofix: automatic on dev pushes (with loop prevention), runs on same-repo PRs, and available manually via workflow_dispatch (fork PRs are skipped)

Why semgrep is not inside quality

quality is designed to fail fast in seconds. semgrep is kept separate because it can take longer and may depend on network/rule downloads. Keeping it separate improves throughput while still making the overall CI fail if semgrep fails.

Why autofix is limited on PRs

Autofix includes loop prevention and can push commits. For security, it runs only on PRs coming from branches in the same repository (fork PRs are skipped).

Autofix (current behavior)

  • Runs automatically on dev pushes (skips commits whose message starts with chore: autofix to avoid loops)

  • Requires PAT_TOKEN to be present as a repository secret to push commits that retrigger workflows

Docker Build Workflow (docker-build.yml)

Builds Docker images, scans them, and publishes them to GHCR.

Trigger events

  • pull_request on main/dev (paths-filtered): build only (no push)

  • push on main/dev (paths-filtered): build + push by SHA, then promote branch tag if CI is green

  • push tag v*: build + push the version tag

  • Workflow-only changes under .github/workflows/** can still trigger this workflow, but heavy jobs are skipped unless runtime/deploy files changed (see changes job in docker-build.yml)

Jobs (current)

  • build (self-hosted docker-heavy):

    • builds the image

    • pushes :sha-<shortsha> on branch pushes

    • runs Trivy scan

  • promote (GitHub-hosted):

    • waits for CI checks on the same commit

    • if CI is green, retags :dev/:main to the corresponding :sha-<shortsha> without rebuilding

    • runs locally on the same machine as staging (no Tailscale/SSH hop)

    • Automated Cleanup: Executes docker image prune after each deployment to keep the server disk healthy (removes unused images older than 7 days).

    • Reliability: The deploy is triggered if CI and build are green, ensuring staging is updated even if minor security warnings are found (while main remains strictly protected).

    • Sentry Integration: Notifies Sentry of new releases using the commit SHA.

Runner selection (self-hosted + GitHub-hosted)

Workflows select runners via repository variables:

  • ACTIONS_RUNNER_LABELS_GH: GitHub-hosted (recommended ["ubuntu-latest"])

  • ACTIONS_RUNNER_LABELS_DOCKER: self-hosted runners with Docker available

  • ACTIONS_RUNNER_LABELS_PY: optional self-hosted runners for Python-only jobs

  • ACTIONS_RUNNER_LABELS_DEPLOY: optional self-hosted runner labels dedicated to deployments

If variables are not set, workflows fall back to ["ubuntu-latest"].

Execution Flow

Pull Request

PR opened/updated
  ↓
CI runs (quality + semgrep + test)
  ↓
Docker Build runs (build only, no push)
  ↓
All required checks green → merge allowed

Main Branch Push

Merge to main
  ↓
CI runs and must be green
  ↓
Docker Build pushes sha-tag and promotes main-tag after CI success
  ↓
Release workflow tags and publishes a GitHub release (main only)

Version Tag

Create tag vX.Y.Z
  ↓
Docker Build builds and pushes vX.Y.Z image tag

Status Checks

Required Checks (main branch)

Add to branch protection rules:

  • ✅ CI / quality

  • ✅ CI / semgrep

  • ✅ CI / test

  • ✅ Docker Build / build

  • ✅ Docs / build-docs

View Status

GitHub UI:
  Settings → Branches → Branch protection rules

Pull Request:
  Shows check status: ✅ or ❌

Debugging

View Workflow Runs

GitHub UI:
  Actions tab → Workflow runs

Check Logs

Click on workflow run → Click on job → View logs

Common issues

  • Quality fails: run ruff check and ruff format locally

  • Semgrep fails: check network/rules download; retry on GitHub-hosted

  • Docker build fails on self-hosted: verify Docker daemon access and disk space

Performance Optimization

Caching

  • uv cache: enabled via astral-sh/setup-uv and keyed on uv.lock

  • Docker layer cache: Buildx uses cache-from/to: type=gha

  • Trivy DB cache: persisted via actions/cache on ~/.cache/trivy

Parallel Jobs

CI jobs run in parallel where possible; Docker Build runs in parallel to CI but promotion/deploy are gated by CI success.

Best Practices

Branch Strategy

  1. Feature branches → PR triggers CI only

  2. dev branch → Passes CI → Docker build

  3. main branch → Stable releases

Semantic Versioning

Tag format: v1.2.3

  • 1 = Major (breaking changes)

  • 2 = Minor (features)

  • 3 = Patch (fixes)

Monitoring

Notifications

  • ✉️ Failed workflow email

  • 🔔 GitHub notifications

  • Slack integration (optional)

Metrics

Track in CI/CD dashboard:

  • Build time

  • Test coverage

  • Deployment frequency

  • Failure rate
