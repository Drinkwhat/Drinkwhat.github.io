API Reference

Questa pagina riassume gli endpoint principali. Per la lista completa e sempre aggiornata, usa:

  • Swagger UI: GET /docs

  • OpenAPI JSON: GET /openapi.json

API IDs: Tutti i path parameter usano UUID opachi (public_id) invece di interi sequenziali. Questo previene attacchi IDOR per enumerazione. Il campo id: int rimane nelle response per compatibilità e sarà rimosso in una versione futura.

Authentication

La sicurezza è basata su Keycloak (OIDC) con token JWT:

TOKEN="<bearer-token>"
curl -H "Authorization: Bearer ${TOKEN}" http://localhost:8000/health

Se un utente ha accesso a più store, può selezionare lo store target tramite:

curl -H "Authorization: Bearer ${TOKEN}" -H "x-target-store: 1" http://localhost:8000/inventory/list

Dettagli: 09-auth.md

Core Headers (Tracing & Multi-tenancy)

Per garantire tracciabilità e isolamento dei dati, il sistema utilizza i seguenti header:

Header

Descrizione

Obbligatorio

Authorization

Bearer token JWT da Keycloak.

Sì

X-Request-ID

ID univoco della richiesta per il tracing. Se non fornito, viene generato dal server.

No

X-Target-Store

ID dello store su cui operare (necessario se l’utente ha accesso a più store).

Inventory Module (/inventory)

POST /inventory/analyze

Analizza una foto di etichetta con OCR + Agent.

Request (multipart/form-data):

curl -H "Authorization: Bearer ${TOKEN}" \
  -F "file=@label.jpg" \
  "http://localhost:8000/inventory/analyze?background=false"

Response (200 OK - Synchronous):

{
  "status": "revision",
  "data": {
    "prodotto": "latte intero",
    "lotto": "L12345",
    "scadenza": "2024-12-31",
    "confidence": 0.95,
    "is_corrected": false
  },
  "warning": null,
  "raw_ocr_debug": "LATTE INTERO L12345 SCAD 31/12/24"
}

Response (200 OK - Asynchronous):

{
  "status": "queued",
  "job_id": "84d72023-..."
}

Errors:

  • 400 Bad Request: File vuoto o dati non validi.

  • 422 Unprocessable Entity: Parsing AI fallito.

  • 502 Bad Gateway: Errore OCR/servizio esterno.

Response (200 OK - In Progress):

{

GET /inventory/list

List batches for the current store with optional filters.

Request:

GET /inventory/list?page=1&page_size=20&expired=false
Authorization: Bearer <token>

Query Parameters (principali):

  • page: Numero pagina (default: 1)

  • page_size: Elementi per pagina (default: 20)

  • product_name: Filter by product name (partial match)

  • is_empty: Filter by empty status (true/false)

  • expired: Filter by expiration status (true=expired)

Response (200 OK):

{
  "status": "success",
  "count": 1,
  "items": [
    {
      "batch_id": 42,
      "product_name": "latte intero",
      "lot_code": "L12345",
      "expiry_date": "2024-12-31",
      "store_id": 1,
      "operator_id": "chef_marco",
      "confidence": 0.95,
      "created_at": "2024-02-05T10:00:00Z",
      "is_empty": false
    }
  ]
}

POST /inventory/batch/{batch_id}/move-storage

Sposta un lotto su un nuovo storage_method_id e ricalcola storage_expiry_at usando la regola (ProductMaster, StorageMethod).

Request:

POST /inventory/batch/42/move-storage
Content-Type: application/json
Authorization: Bearer <token>

{
  "version": 1,
  "storage_method_id": 12
}

POST /inventory/batch/{batch_id}/complete

Segna un lotto come concluso/esaurito. Valorizza batches.finished_at e crea un record append-only in batch_events.

Documentazione completa del flusso (preparazione + lotto): 18-production-completion.md

Nota: richiede un body con reason (motivo standardizzato) e notes opzionali.

Setup Module (/setup)

Solo admin. Tutti gli endpoint /setup richiedono il permesso ADMIN_ALL (ruolo admin) e sono disabilitati in ambiente production (rispondono 404).

POST /setup/seed-demo

Popola lo store demo (id=1) con un dataset completo e testabile in un’unica chiamata: cataloghi (categorie, unità di misura, metodi di conservazione), prodotti con regole di conservazione, lotti a scadenza futura/imminente/passata e vuoti, ricette con ingredienti, preparazioni eseguibili e notifiche.

Query params:

  • reset (bool, default false) — se true cancella lo store demo e lo rigenera da zero (stato pulito ad ogni esecuzione). Senza reset l’operazione è idempotente e non reinserisce dati se lo store esiste già.

Request:

POST /setup/seed-demo?reset=true
Authorization: Bearer <admin_token>

Response (200 OK):

{
  "message": "Store demo popolato con dataset completo.",
  "store_id": 1,
  "reseeded": true,
  "counts": {
    "categories": 8,
    "units_of_measure": 2,
    "storage_methods": 3,
    "products": 15,
    "batches": 30,
    "recipes": 3,
    "recipe_ingredients": 11,
    "preparations": 4,
    "preparation_items": 12,
    "notifications": 5
  }
}

Se chiamato senza reset su store già popolato risponde 200 con "reseeded": false. Per chef/manager risponde 403; in production risponde 404.
