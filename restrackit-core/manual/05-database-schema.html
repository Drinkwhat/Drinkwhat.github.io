Table Definitions

stores

Restaurant or chain locations. Supports multi-tenancy.

CREATE TABLE stores (
  id INTEGER PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  address VARCHAR(500),
  city VARCHAR(100),
  phone VARCHAR(20),
  email VARCHAR(255),
  is_active BOOLEAN DEFAULT TRUE,
  notify_expiring_today BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

Note: tabella tenant. Con RLS attivo, ogni richiesta lavora su un solo store_id.

storage_methods

Catalogo dei metodi di storage definito a livello store.

CREATE TABLE storage_methods (
  id INTEGER PRIMARY KEY,
  store_id INTEGER NOT NULL REFERENCES stores(id),
  name VARCHAR(100) NOT NULL,
  deleted_at TIMESTAMP WITH TIME ZONE,
  version INTEGER NOT NULL DEFAULT 1,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  CONSTRAINT _store_storage_method_name_uc UNIQUE (name, store_id)
);

product_storage_rules

Regole per definire una durata (delta time) per la coppia (ProductMaster, StorageMethod).

CREATE TABLE product_storage_rules (
  id INTEGER PRIMARY KEY,
  store_id INTEGER NOT NULL REFERENCES stores(id),
  product_master_id INTEGER NOT NULL REFERENCES products_master(id),
  storage_method_id INTEGER NOT NULL REFERENCES storage_methods(id),
  duration_days INTEGER NOT NULL DEFAULT 0,
  duration_hours INTEGER NOT NULL DEFAULT 0,
  deleted_at TIMESTAMP WITH TIME ZONE,
  version INTEGER NOT NULL DEFAULT 1,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  CONSTRAINT _store_product_storage_rule_uc UNIQUE (store_id, product_master_id, storage_method_id)
);

categories

Catalogo categorie prodotto, scoped per store. Soft-delete + optimistic locking.

CREATE TABLE categories (
  id INTEGER PRIMARY KEY,
  store_id INTEGER NOT NULL REFERENCES stores(id),
  name VARCHAR(100) NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  deleted_at TIMESTAMP WITH TIME ZONE,
  version INTEGER NOT NULL DEFAULT 1,
  CONSTRAINT _store_category_name_uc UNIQUE (name, store_id)
);

units_of_measure

Catalogo unità di misura, scoped per store. Soft-delete + optimistic locking.

CREATE TABLE units_of_measure (
  id INTEGER PRIMARY KEY,
  store_id INTEGER NOT NULL REFERENCES stores(id),
  name VARCHAR(100) NOT NULL,
  symbol VARCHAR(20) NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  deleted_at TIMESTAMP WITH TIME ZONE,
  version INTEGER NOT NULL DEFAULT 1,
  CONSTRAINT _store_uom_symbol_uc UNIQUE (symbol, store_id)
);

products_master

Catalogo prodotti (ingredienti e semilavorati) per store.

CREATE TABLE products_master (
  id INTEGER PRIMARY KEY,
  store_id INTEGER NOT NULL REFERENCES stores(id),
  name VARCHAR(255) NOT NULL,
  category_id INTEGER NOT NULL REFERENCES categories(id),
  allergen_mask INTEGER NOT NULL DEFAULT 0,
  deleted_at TIMESTAMP WITH TIME ZONE,
  version INTEGER NOT NULL DEFAULT 1,
  CONSTRAINT _store_product_name_uc UNIQUE (name, store_id)
);

Note: category_id è FK a categories. Il campo category (stringa) è esposto via property ORM per compatibilità API.

Allergeni: allergen_mask è una bitmask (0 = nessun allergene). La lista “slug” è esposta via API (GET /products/allergens).

batches

Lotti di magazzino (tracciabilità + scadenze).

CREATE TABLE batches (
  id INTEGER PRIMARY KEY,
  store_id INTEGER NOT NULL REFERENCES stores(id),
  product_master_id INTEGER NOT NULL REFERENCES products_master(id),

  lot_code VARCHAR(255) NOT NULL,
  expiry_date DATE NOT NULL,
  current_storage_method_id INTEGER REFERENCES storage_methods(id),
  storage_expiry_at TIMESTAMP WITH TIME ZONE,
  operator_id VARCHAR(255) NOT NULL,

  confidence FLOAT NOT NULL DEFAULT 0.0,
  is_empty BOOLEAN NOT NULL DEFAULT FALSE,

  finished_at TIMESTAMP WITH TIME ZONE,
  deleted_at TIMESTAMP WITH TIME ZONE,
  version INTEGER NOT NULL DEFAULT 1,

  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  CONSTRAINT check_confidence_range CHECK (confidence >= 0 AND confidence <= 1)
);

Indexes:

  • INDEX on store_id

  • INDEX on product_master_id

  • INDEX on lot_code

  • INDEX on operator_id

  • INDEX on is_empty

  • INDEX on created_at

  • INDEX on current_storage_method_id

  • INDEX on storage_expiry_at

  • INDEX on (store_id, product_master_id, created_at) per post-processing OCR e storico lotti

batch_storage_moves

Storico dei trasferimenti di uno specifico lotto tra metodi di storage, con scadenza calcolata.

CREATE TABLE batch_storage_moves (
  id INTEGER PRIMARY KEY,
  store_id INTEGER NOT NULL REFERENCES stores(id),
  batch_id INTEGER NOT NULL REFERENCES batches(id),
  from_storage_method_id INTEGER REFERENCES storage_methods(id),
  to_storage_method_id INTEGER NOT NULL REFERENCES storage_methods(id),

Data Integrity

  • Multi-Tenancy: All queries MUST filter by store_id to prevent data leaks between restaurants. This is enforced at the database level using Row Level Security (RLS).

Row Level Security (RLS)

To ensure strict isolation between tenants (stores), the database uses PostgreSQL Row Level Security.

  1. Mechanism: Each table containing tenant data has RLS enabled and forced (FORCE ROW LEVEL SECURITY). This ensures isolation even when the application connects as the table owner.

  2. Policy: Access is restricted based on the PostgreSQL session variable app.current_store_id.

  3. Context Injection: The application sets this variable at the start of each transaction using:

    SET LOCAL app.current_store_id = 'target_store_id';

  4. Enforced Tables:

    • stores

    • categories

    • units_of_measure

    • products_master

    • storage_methods

    • product_storage_rules

    • batches

    • batch_storage_moves

    • recipes

    • preparations

    • recipe_ingredients (via subquery su recipes)

    • preparation_items (via subquery su preparations)

    • notifications

Important: While RLS is active, application-level filters (.filter(Model.store_id == id)) should still be used as a “Defense in Depth” measure.

Constraints (principali):

  • UNIQUE(name, store_id) per products_master, recipes, storage_methods, categories

  • UNIQUE(symbol, store_id) per units_of_measure

  • UNIQUE(store_id, product_master_id, storage_method_id) per product_storage_rules

  • check_confidence_range su batches.confidence (0..1)

  • Soft-delete con deleted_at e concorrenza ottimistica con version (SQLAlchemy version_id_col)
